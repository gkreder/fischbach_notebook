Fixing Bokeh Data Column Mismatch

2017-05-26
Attachments:
Cas9_REC_heatmap.html , Cas9_REC_pruned.html

CAS 05-26-17

Looking at the bokeh html output on the CAS heatmap, there's a weird column insertion. There's a strip of random PFAM terms between Cas9_REC and HNH_4. When I go into one of the pfd files and look at the locations, there shouldn't be any PFAM insertion there...

I think I might have to quickly reorganize the data structure. The real issue is the block data structure I think. It should be organized by lines... I'm going to take a quick look at the make_blocks.py file and see if I can't make it spit out a simpler lines data structure...

Rearranging data structure

I've repurposed make_blocks.py into a new script called make_lines.py that spits out a data file called \<pfam_term>_lines.pkl. This file is a simple list of list of pfam objects (that should self-contain all their relevant information). Each list of pfam objects represents a line in the plot.

Data_prep.py update

I now have a function called lines_to_2d(pfam_target) that spits out a data_dict. The entries in the dict are largely the same as before, minus the shape coordinates for rectangles and arrowheads, will have to revisit that later. But the creation of the dict is much cleaner now. I've also passed along responsibility for creating the ColumnDataSource to the plotting function itself which now takes the data_dict as an argument rather than the the ColumnDataSource or DataFrame. The reason for this is that if I take the data_dict into the plotting function I can still do more manipulations to it (e.g. add shape coordinates) before turning into a DataFrame/ColumnDataSource. I think this will help internalize more plotting-specific things to the figures.py code.

Erroneous Column

After doing this cleanup the weird column insertion between Cas9_REC and HNH_4 is gone. I'm going to try to re-run the pruning and exclusion code to see what pops out.
